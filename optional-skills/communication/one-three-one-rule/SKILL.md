# 1-3-1 沟通规则

用于当任务有多个可行方法且用户需要明确建议时的结构化决策格式。生成简洁的问题框架、三个带权衡的选项，以及推荐路径的可执行计划。

## 何时使用

- 用户明确要求"1-3-1"响应。
- 用户对于技术决策说"给我选项"或"我有哪些选择"。
- 任务有多个可行的方法且有意义的权衡（架构、工具、迁移策略）。
- 用户需要可以转发给团队或利益相关者的提案。

对于简单问题（只有一个明显答案）、调试会话或用户已决定方法的任务，**不要**使用。

## 程序

1. **问题**（一句话）
   - 用一句简洁的话陈述核心决策或期望结果。
   - 专注于*什么*，而非*如何* — 不包含实现细节、工具名称或特定技术。
   - 保持简洁。如果您需要"和"，您是在描述两个问题。

2. **选项**（恰好三个）
   - 展示三个标记为 A、B、C 的不同可行方法。
   - 每个选项都有简要描述、优点和缺点。
   - 选项应代表真正不同的策略，而非同一方法的微小变化。

3. **建议**（一个选项）
   - 陈述您建议哪个选项及原因，基于用户的上下文和优先级。
   - 要直接 — 这是您的专业判断，不是含糊其辞。

4. **完成定义**
   - 列出推荐选项的特定成功标准。
   - 这些是具体、可验证的结果 — 不是模糊的愿望。
   - 如果用户选择不同选项，相应修改此部分。

5. **实施计划**
   - 执行推荐选项的具体步骤。
   - 在适当时包含特定命令、工具或操作。
   - 如果用户选择不同选项，相应修改此部分。

## 验证

- 响应恰好有一个问题句子。
- 响应恰好有三个选项（A、B、C），每个都有优缺点。
- 响应有一个单一建议，选择一个选项并给出理由。
- 完成定义和实施计划与推荐的选项一致。
- 如果用户选择不同选项，建议、完成定义和实施计划相应更新。

## 示例

用户："给我添加重试逻辑到我们的 API 客户端的 1-3-1。"

代理：

**问题：** API 调用在负载下间歇性失败，需要优雅的重试处理。

**选项：**

- **选项 A：在共享工具中使用指数退避加抖动。**
  优点：简单、广泛理解的模式；所有调用者共享单一实现。
  缺点：无熔断；调用者在重试期间仍会阻塞。

- **选项 B：重试库（tenacity）作为每个端点调用的装饰器。**
  优点：每个端点可配置；内置支持熔断和速率限制。
  缺点：增加依赖；装饰器模式可能掩盖控制流。

- **选项 C：基于队列的异步重试与死信处理。**
  优点：非阻塞；失败的调用被跟踪并可重放；扩展性好。
  缺点：显著增加复杂性；需要队列后端；对于低容量服务过度设计。

**建议：** 选项 A。调用量不需要队列基础设施，共享工具加指数退避加抖动干净地覆盖失败模式，无需新依赖。

**完成定义：**
- `api_retry()` 工具处理瞬态 HTTP 错误（429、502、503、504）。
- 最多重试 3 次，指数退避加抖动。
- 所有外部 API 调用通过重试工具路由。
- 每次重试尝试记录端点和尝试次数。
- 测试覆盖重试耗尽、立即成功和瞬态然后成功场景。

**实施计划：**
1. 创建 `utils/api_retry.py`，包含可配置的最大重试次数、基本延迟和可重试状态码。
2. 使用 `random.uniform(0, base_delay)` 添加抖动以防止雷鸣般的群效应。
3. 使用重试工具包装 `api_client.py` 中的现有 API 调用。
4. 添加单元测试，模拟每个重试场景的 HTTP 响应。
5. 对 flaky 端点 mock 进行简单压力测试验证。
