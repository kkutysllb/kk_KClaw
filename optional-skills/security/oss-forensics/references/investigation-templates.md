# 调查模板

常见供应链攻击场景的预建假设和调查模板。
每个模板包括：攻击模式、要收集的关键证据和假设启动器。

---

## 模板 1：维护者账户妥协

**模式**：攻击者获得合法维护者账户的访问权限（网络钓鱼、凭据填充），并使用它推送恶意代码、创建带后门的发布或泄露 CI 秘密。

**真实案例**：XZ Utils (2024)、Codecov (2021)、event-stream (2018)

**要收集的关键证据**：
- [ ] 在正常工作时间/时区之外从维护者账户的推送事件
- [ ] 添加新依赖项、混淆代码或修改构建脚本的提交
- [ ] 可疑推送后立即创建发布（最大化包分发）
- [ ] 添加未知协作者的 MemberEvent（攻击者添加备份访问）
- [ ] 具有意外秘密访问或类似泄露行为的 WorkflowRunEvent
- [ ] 账户登录位置更改（检查社交媒体、会议演讲以进行验证）

**假设启动器**：
```
[HYPOTHESIS] 演员 <HANDLE> 的账户在 <DATE> 左右被入侵，
基于异常提交时间 [EV-XXXX] 和地理访问模式 [EV-YYYY]。
```
```
[HYPOTHESIS] 发布 <VERSION> 由被入侵的账户发布，以将恶意代码推送给下游用户，
证据是恶意提交 [EV-XXXX] 在发布前 <N> 小时被添加 [EV-YYYY]。
```

## 模板 2：恶意依赖注入

**模式**：受信任的包被修改以包含恶意代码在依赖项中，或新的恶意依赖项被注入到现有包中。

**要收集的关键证据**：
- [ ] 可疑提交前后 `package.json`/`requirements.txt`/`go.mod` 的 diff
- [ ] 新依赖项的发布 时间戳与注入提交时间戳的对比
- [ ] 新依赖项是否存在于 npm/PyPI 上及其所有者是谁
- [ ] 注入依赖项代码中的任何混淆模式
- [ ] 安装时执行的脚本（`postinstall`、`setup.py` 等）

**假设启动器**：
```
[HYPOTHESIS] 提交 <SHA> [EV-XXXX] 引入了依赖项 <PACKAGE@VERSION>，
这似乎是由演员 <HANDLE> [EV-YYYY] 发布的恶意包，
旨在安装期间执行 <BEHAVIOR>。
```

## 模板 3：CI/CD 管道注入

**模式**：攻击者修改 GitHub Actions 工作流以窃取秘密、泄露代码或将恶意产物注入构建输出。

**要收集的关键证据**：
- [ ] 可疑时期前后所有 `.github/workflows/*.yml` 文件的 diff
- [ ] 由修改后的工作流触发的 WorkflowRunEvents
- [ ] 添加到工作流步骤的任何 `curl`、`wget` 或网络调用
- [ ] 引用 `secrets.*` 的新的或修改的 `env:` 部分
- [ ] 修改后的工作流运行产生的产物

**假设启动器**：
```
[HYPOTHESIS] 工作流文件 <FILE> 在提交 <SHA> [EV-XXXX] 中被修改，
通过 <METHOD> 泄露仓库秘密，如添加的网络调用模式 [EV-YYYY] 所证明。
```

## 模板 4：Typosquatting / 依赖混淆

**模式**：攻击者注册一个与流行包名称相似的包（或内部包名称）以拦截拼写错误的用户的安装。

**要收集的关键证据**：
- [ ] 可疑包在注册商上的注册时间戳
- [ ] 包内容：它是否包含恶意代码或是一个存根？
- [ ] 可疑包的下载统计
- [ ] 可能被针对的内部包名称（如果是私有仓库范围）
- [ ] 恶意包在元数据中对合法包的任何引用

**假设启动器**：
```
[HYPOTHESIS] 包 <MALICIOUS_NAME> 在 <DATE> [EV-XXXX] 注册，
以 typosquat <LEGITIMATE_NAME>，针对拼写包名称的用户。
该包包含 <BEHAVIOR> [EV-YYYY]。
```

## 模板 5：强制推送历史重写（证据擦除）

**模式**：在检测到恶意提交后（或在更广泛注意之前），攻击者强制推送以从分支历史中删除恶意提交。

**检测是关键** — 此模板专注于证明擦除发生。

**要收集的关键证据**：
- [ ] 具有 `distinct_size=0` 的 GH Archive PushEvent（强制推送指标）[EV-XXXX]
- [ ] 强制推送前提交的 SHA（来自 GH Archive `payload.before`）
- [ ] 通过直接 URL 或 `git fetch origin SHA` 恢复被擦除的提交
- [ ] 擦除前提交页面的 Wayback Machine 快照
- [ ] git 日志中的时间线间隙（归档中可见 N 个提交但当前仓库中 M < N）

**假设启动器**：
```
[HYPOTHESIS] 演员 <HANDLE> 在 <DATE> [EV-XXXX] 强制推送分支 <BRANCH>，
以擦除包含 <MALICIOUS_CONTENT> 的提交 <SHA> [EV-YYYY]。
被擦除的提交通过 <METHOD> [EV-ZZZZ] 恢复。
```

## 跨调查检查清单

无论模板如何，适用于每个调查：

- [ ] 检查所有贡献者中最近创建的账户（恶意活动时不足 30 天）
- [ ] 检查是否有任何维护者账户在期间更改了电子邮件（账户接管的迹象）
- [ ] 验证可疑提交上的 GPG 签名与已知维护者密钥匹配
- [ ] 检查仓库是否在事件附近更改了所有者或转移到组织
- [ ] 在恶意提交后立即查找"清理"提交（掩盖模式）
- [ ] 检查同一作者的相關包/仓库是否有类似模式
